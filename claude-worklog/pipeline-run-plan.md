# OurStory — Pipeline Execution Plan
*Written: 2026-03-01*

This document covers the phased run strategy, time estimates, and distributed
architecture for populating OurStory's database from Wikidata.

---

## Architecture Constraint: The bz2 Bottleneck

The Wikidata dump (`latest-all.json.bz2`, ~100 GB) is a **single bz2 stream —
not splittable**. Unlike the Wikipedia XML multistream dump, there is no index
file for random access. This means:

- `bzcat` is single-threaded by nature; Spark cannot parallelize decompression
  of a single bz2 file
- If you point Spark at the bz2 directly, one task handles the entire file

**Resolution:** Before any Dataproc work, decompress the dump and re-split it
into ~20 gzip chunks on a single high-CPU VM. gzip files are also not splittable,
but by having 20 separate files, Spark assigns one task per file — giving 20×
parallelism for all downstream processing.

```
single .bz2 file (100 GB)
    ↓  bzcat + split (1 VM, ~60–90 min)
20× .ndjson.gz chunks (~5 M lines each, ~5 GB each)
    ↓  Dataproc (20+ workers, ~20 min)
spatiotemporal-entities.parquet
```

Everything downstream of the chunk stage parallelizes normally.

## Code Structure (Do First)

Before any run, refactor the extraction logic out of `validate-sample.py` into
standalone modules. The same logic should run identically in plain Python (local)
and inside a Spark `mapPartitions` call (GCP), with no duplication.

```
pipeline/
  __init__.py
  extract.py          ← extract_event(), parse_wikidata_time(), map_categories()
                        get_claim_value(), get_coord(), etc. — pure Python, no deps
  taxonomy.py         ← SPARQL taxonomy enumeration, save/load event_classes.json
  load_postgres.py    ← psycopg2-based bulk COPY loader

  run_local.py        ← API-based run: Wikidata API + Wikipedia REST API, plain Python
  run_spark.py        ← Dataproc/PySpark run: reads chunks, calls extract.py functions
  run_taxonomy.py     ← CLI entry point for Step 0

  data/
    event_classes.json   ← generated by run_taxonomy.py
```

`extract.py` is the core. It has zero GCP/Spark dependencies. `run_local.py` and
`run_spark.py` are thin wrappers that handle I/O and parallelism differently, but
call the same functions.

## Phased Run Strategy

### Run 1 — Local API (500 events) · Est. 2 hours

**Goal:** End-to-end pipeline working locally. Real data in Docker Postgres.
Frontend updated with more than the hand-curated seed data.

**Method:** Wikidata SPARQL (5 × 100-event batches) → batch Wikidata API fetch
→ Wikipedia REST API for summaries → extract → load to local Postgres.

Use Wikipedia REST API (`/api/rest_v1/page/summary/{title}`) instead of DBpedia
here — it returns clean first-paragraph text per request, no RDF parsing needed.
DBpedia bulk extract is only worth it at scale.

**Steps:**
1. Refactor `validate-sample.py` → `pipeline/extract.py` + `pipeline/run_local.py`
2. Write SPARQL pagination loop: fetch 500 QIDs across 5 categories (battles,
   natural disasters, revolutions, elections, explorations)
3. Batch-fetch entities via Wikidata API (50 QIDs/request → 10 requests)
4. Fetch Wikipedia summaries (1 request/event, ~0.5 s/request → ~4 min for 500)
5. Run extraction + category mapping + location QID resolution
6. Load to Docker Postgres
7. Export updated `seed.geojson` via `scripts/export-geojson.ts`
8. Verify in frontend: events visible on map with correct years, categories, locations

**Expected coverage for 500 events:**
- Dates: ~480/500 (96%) — well-typed events are well-structured
- Coords (P625): ~200/500 (40%)
- Location QID (P276): ~220/500 (44%)
- No location: ~80/500 (16%) — needs LLM, skip for now
- Categories from P31: ~450/500 (90%)

**Rate limiting:** Wikipedia REST API: 200 requests/second per client, no issue.
Wikidata API: 50 QIDs/request, ~1 request/second safe limit → 10 seconds for 500.

**No GCP required.**

### Run 2 — Local API scaled (5,000 events) · Est. 4 hours

**Goal:** Stress-test the category mapping and location QID resolution. Find
gaps in `WIKIDATA_TO_CATEGORY`. Generate a richer seed dataset for the frontend.

**Method:** Expand Run 1 to 5,000 events across all major subclasses. SPARQL
with multiple targeted category queries.

**SPARQL strategy — query by specific subclass, not the full tree:**
```sparql
# e.g. all battles with coordinates and English Wikipedia article
SELECT ?qid WHERE {
  ?qid wdt:P31/wdt:P279* wd:Q178561 .   # battle subclass tree
  ?qid wdt:P585|wdt:P580 ?date .          # has a date
  ?article schema:about ?qid ;
           schema:isPartOf <https://en.wikipedia.org/> .
}
LIMIT 1000
```
Run 5 such queries (battles, wars, elections, disasters, explorations) →
5,000 QIDs total. Each query returns fast (< 60 s) because it's type-restricted.

**New work vs Run 1:**
- Wikidata API: 100 requests (50 QIDs each) → ~2 minutes
- Wikipedia summaries: ~5,000 requests at ~0.5 s = ~42 minutes (main runtime)
  - Parallelize with `concurrent.futures.ThreadPoolExecutor(max_workers=8)` → ~5 min
- Location QID resolution: collect all unique P276 QIDs, batch-resolve via API
- Category gap analysis: log any P31 QIDs not in `WIKIDATA_TO_CATEGORY`, add them
- Load to Postgres, export GeoJSON, verify frontend

**No GCP required.**

### Run 3 — GCP partial dump (50,000–100,000 events) · Est. 1 day

**Goal:** Validate the Dataproc pipeline end-to-end. Confirm GCS, Dataproc,
and Postgres connectivity. Validate the DBpedia abstract join at scale.
Catch any infrastructure bugs before committing to the full 100 GB run.

**Method:** Take only the first ~3 GB of the bz2 dump. This is achieved by
decompressing a partial stream:

```bash
# On a GCP VM (e2-standard-4, ~$0.13/hr):
bzcat latest-all.json.bz2 | head -n 5000000 > wikidata-partial.ndjson
# Wikidata entities are ~1–3 KB each; 5M lines ≈ 5–15 GB uncompressed
# Contains ~50K–100K events depending on how early in the dump they appear
gzip wikidata-partial.ndjson
gsutil cp wikidata-partial.ndjson.gz gs://ourstory-data/wikidata/
```

Then run the full Spark pipeline on that single file.

**Also do in Run 3:**
- Download DBpedia short abstracts to GCS and parse to NDJSON lookup
- Run the DBpedia join step (Step 5)
- Validate that abstracts are joining correctly on `wikipedia_title`

**Infrastructure to set up:**
- GCS bucket: `ourstory-data`
- Cloud SQL (Postgres 16) or a large VM with Postgres
- Dataproc cluster: 1 master (n2-standard-4) + 4 workers (n2-standard-8, preemptible)
- Service account with GCS + Dataproc + Cloud SQL permissions

**Expected runtime:**
- Partial bz2 decompress + head (VM): ~5 min
- Tier 1 filter (Dataproc, 4 workers): ~8 min
- Tier 2 event filter: ~2 min
- Structured extraction: ~3 min
- DBpedia join: ~5 min
- Load to Cloud SQL: ~5 min
- **Total Dataproc time: ~25 min**

**Cost:** Dataproc (4 workers × n2-standard-8 preemptible × 0.5 hr): ~$0.80.
VM for decompression: ~$0.10. Cloud SQL: ~$0.05/hr.

### Run 4 — Full GCP pipeline, no LLM (2M events) · Est. 8–10 hours

**Goal:** All ~2M Wikidata events extracted, structured, and loaded. No LLM
enrichment yet — location and category gaps will exist but that's acceptable.
This is the first complete database.

#### Stage A: Decompress + Chunk (1 VM, sequential, ~90 min)

```bash
# c2-standard-16 VM (~$0.65/hr):
bzcat latest-all.json.bz2 \
  | split -l 5000000 --filter='gzip > $FILE.ndjson.gz' chunk_

# Produces: chunk_aa.ndjson.gz ... chunk_as.ndjson.gz (~19 files × ~5M lines)
gsutil -m cp chunk_*.ndjson.gz gs://ourstory-data/wikidata/chunks/
```

Estimated time: 60–90 min for decompression (bz2 → stdout: ~25 MB/s
on a 100 GB file = 67 min) + ~15 min for gzip + upload.

#### Stage B: Tier 1 Structural Filter (Dataproc, parallel, ~25 min)

Dataproc reads all 19 chunk files simultaneously. Each worker processes one
file at a time. 19 files / 10 workers = ~2 passes.

```python
# run_spark.py (Tier 1)
raw = spark.read.text("gs://ourstory-data/wikidata/chunks/*.ndjson.gz")
# Spark assigns one partition per file → 19 partitions → 19 parallel tasks
filtered = raw.rdd.filter(lambda row: has_spatiotemporal_data(row.value))
filtered.saveAsTextFile("gs://ourstory-data/intermediate/spatiotemporal/",
                        compressionCodecClass="org.apache.hadoop.io.compress.GzipCodec")
```

Cluster: 1 master (n2-standard-4) + 10 workers (n2-standard-8 preemptible).
Each worker processes ~65 MB/s compressed input → 19 files / 10 workers ×
~3 min/file = ~6 min. Add startup overhead: ~25 min total.

Output: ~5–15 GB spatiotemporal-entities (1–3M records).

#### Stage C: Tier 2 + Extraction + DBpedia Join (Dataproc, ~20 min)

Reads the small intermediate file (5–15 GB). All steps run on the same cluster
without restarting.

```python
# run_spark.py (Tier 2 + extraction, chained)
entities = spark.read.text("gs://ourstory-data/intermediate/spatiotemporal/")

# Tier 2: event filter
event_classes = broadcast(load_event_classes())
events_raw = entities.rdd.filter(lambda r: is_event_entity(r, event_classes.value))

# Structured extraction
events_df = events_raw.map(extract_event).toDF()

# DBpedia join
abstracts = spark.read.json("gs://ourstory-data/dbpedia/abstracts-en.ndjson")
enriched = events_df.join(abstracts, on="wikipedia_title", how="left")

enriched.write.parquet("gs://ourstory-data/output/events-enriched/")
```

Estimated time: ~20 min (mostly Tier 2 filter + extraction).

#### Stage D: Location QID Resolution (~30 min)

Collect all distinct `location_qid` values from the Parquet output.
Batch-resolve to coordinates via Wikidata API. This runs on a single VM,
not Dataproc — the API is the bottleneck, not compute.

```python
# Rough estimate:
# ~800K events have P276; ~50K unique location QIDs
# 50K QIDs / 50 per request = 1,000 API requests
# At 1 req/second (safe rate limit) = 17 minutes
# Add parallel workers (5 threads): ~3–4 minutes
```

Output: `location_coords.json` — `{QID: [lat, lon], ...}`.

#### Stage E: Category Mapping (~5 min)

Apply static `WIKIDATA_TO_CATEGORY` map. Flag events needing LLM.
This is a pure Python script on a single VM — no Spark needed.

#### Stage F: Load to PostgreSQL (~45 min)

Use Postgres `COPY` (not INSERT) for bulk loading:
1. Cities first (Step 8a): ~500K rows
2. Events second (Step 8b): ~2M rows, with location QID → city UUID resolution

#### Stage G: Export + Validate

```sql
SELECT COUNT(*), COUNT(lat), COUNT(wikipedia_summary), COUNT(categories[1])
FROM events;
```

Expected counts post-Run 4:
| Metric | Expected |
|---|---|
| Total events loaded | ~800K–1.2M (dated events with enwiki) |
| With coordinates | ~400K–600K |
| With Wikipedia summary | ~700K–1M |
| With category assigned | ~600K–900K |
| Needing LLM location | ~300K–500K |
| Needing LLM category | ~100K–200K |

### Run 5 — LLM Enrichment (overnight batch) · Est. 4–24 hours

**Goal:** Fill remaining location and category gaps using Claude API.

**Method:** Claude Batch API (asynchronous). Submit up to 10,000 requests per
batch, results returned within 24 hours. Can submit all batches simultaneously.

**Location assignment (Bucket 3):**
- ~400K events with no P625 and no resolvable P276
- Each request: event title + summary → JSON `{location_name, lat, lon, confidence, level}`
- Model: claude-haiku-4-5 (fastest, cheapest, sufficient for geocoding)
- Token estimate: ~120 input tokens + ~40 output tokens = ~160 tokens/event
- Cost: 400K × 120 × $0.80/MTok (input) + 400K × 40 × $4/MTok (output)
  = $38.40 + $64 = **~$100 total**
- Batches: 400K / 10K per batch = 40 batches, submitted in parallel
- Wall-clock time: 1–24 hours (batch API SLA)

**Category assignment (Layer 2):**
- ~150K events typed only as "historical event" or "occurrence"
- Each request: title + summary → one of 12 category labels
- Token estimate: ~80 input + ~10 output = ~90 tokens/event
- Cost: 150K × 90 × avg $1/MTok = **~$13 total**
- Wall-clock time: 1–4 hours

**Total LLM cost estimate: ~$110–130.**

After enrichment, reload enriched rows to Postgres.

## Key Decisions

**Why not stream from bz2 directly to Spark?**
A single bz2 file forces a single Spark task. With 90M entities at ~1 KB each
= 90 GB to process in one task = ~4–6 hours on one worker. The chunk strategy
cuts that to ~20 minutes across 10 workers. Worth the 90-minute upfront cost.

**Why Claude Haiku for LLM enrichment?**
Location assignment and category classification are simple structured tasks — not
complex reasoning. Haiku is 10× cheaper than Sonnet and fast enough. Switch to
Sonnet only if quality validation shows meaningful degradation.

**Why not use Spark for location QID resolution?**
The Wikidata API is rate-limited at the IP level. Running 80 Spark workers all
hammering the same API endpoint would result in 429s and failed retries. A single
VM with 5 controlled threads is the right tool. The total unique location QID set
is ~50K — trivial to process sequentially.

**Why load cities before events?**
The `events.location_id` column is a FK to `cities.id`. Cities must exist before
any event referencing them can be inserted. Build a `{wikidata_qid → uuid}` map
in memory during the city load, pass it to the event load step.
